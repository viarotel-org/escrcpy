## PR 草稿

### 建议标题

支持启动 Android 多用户/分身应用，并为应用提供独立的横屏副屏启动选项

### 概要

这个改动主要增强了现有的“启动 APP”流程，面向的是依赖 Android 多用户、应用分身、副屏多开的使用场景。

本次改动增加了两项比较实用的能力：

1. 识别并启动 Android 次级用户中的应用
2. 为单个应用提供独立的横屏副屏启动能力，并兼容桌面快捷方式

同时也顺手修复了一个配套的下拉层显示问题：

3. 首次打开应用启动下拉时，异步加载完成后主动重新计算弹层位置，避免出现偏移或显示不全

我已经在一台小米 / Redmi 设备上完成了实际测试，同一个游戏安装在多个 Android 用户空间后，可以同时启动到多个 scrcpy 虚拟显示中。

### 问题背景

原有的应用启动流程对于主用户中的普通应用是可用的，但在多开 / 分身场景下有两个明显限制：

1. 安装在 Android 次级用户中的应用，无法通过现有应用列表正常发现。
2. 通过 `new display` 启动副屏时，虚拟显示默认继承主屏的纵向比例，导致横屏游戏可能仍然先启动在竖屏副屏里。

这个问题在下面这些用户场景里会比较明显：

- 使用 MIUI / HyperOS 分身空间或多用户空间
- 希望把同一个游戏同时开在多个副屏
- 更依赖桌面快捷方式一键启动，而不是每次都重新进菜单选择

### 这个改动做了什么

#### 1. 支持扫描 Android 次级用户里的可启动应用

应用选择器现在会补充扫描：

- `pm list users`
- `pm list packages -3 --user <id>`
- `cmd package query-activities --brief --components --user <id> -a android.intent.action.MAIN -c android.intent.category.LAUNCHER`

这样可以把分身 / 次级用户里的应用一起显示到启动列表中，并保留对应的 `userId` 和启动 `activity` 信息。

#### 2. 启动分身应用时，明确指定用户和显示器

针对次级用户中的应用，启动流程改为使用：

- `am start-activity --user <id> --display <displayId> -n <component>`

这样可以确保应用：

- 在正确的 Android 用户空间中启动
- 被准确投放到刚创建的虚拟显示里

#### 3. 在启动下拉区域里，为每个应用增加独立的横屏勾选项

在应用下拉列表的操作区域中，新增了一个勾选框，位置放在现有“创建快捷方式 / 主屏启动”操作旁边。

行为如下：

- 不勾选：保持当前默认行为
- 勾选后：这个应用在副屏启动时，优先按横屏虚拟显示创建

这个勾选状态会按“设备 + 应用项”分别保存，用户不需要每次重新设置。

#### 4. 修复应用启动下拉首次打开时的定位偏移问题

由于应用列表包含异步加载的内容，首次点击打开下拉时，弹层有可能先按旧尺寸完成定位，等列表真正变宽或变高以后，位置没有立即刷新，表现为：

- 首次打开时弹层位置偏移
- 右侧操作区域显示不全
- 再打开一次后恢复正常

这次在下拉组件中补充了两类处理：

- 首次显示且异步列表加载完成后，主动触发一次弹层位置重算
- 当列表条目数量或搜索结果变化时，如果弹层仍处于打开状态，则继续同步更新位置

这样可以让首次点击时的显示结果更稳定，不需要靠第二次打开才能恢复正常。

#### 5. 创建桌面快捷方式时，保留横屏启动参数

如果某个应用当前勾选了横屏模式，那么为它创建桌面快捷方式时，会自动附带：

```text
--landscape=1
```

这样桌面一键启动的行为就和界面里手动点击启动保持一致。

#### 6. 让打包版应用也能接收并透传 landscape 参数

打包后的 Electron 应用现在会把 `landscape` 启动参数透传经过单实例生命周期桥接。

这样即使主程序已经在运行，再次通过命令行或桌面快捷方式启动，也能把这次启动请求正确转发到当前实例。

#### 7. 用“横向尺寸创建副屏”代替“启动后再旋转”

当副屏启动请求带有 `landscape=1` 时，代码会先读取：

- `wm size`
- `wm density`

然后用下面的方式创建 scrcpy 虚拟显示：

```text
--new-display=<max(width,height)>x<min(width,height)>/<density>
```

也就是直接以横向分辨率创建新的副屏，而不是等应用起来以后再尝试旋转。

这种方式更稳定，原因是虚拟显示本身从创建开始就是横屏尺寸，横屏游戏更容易直接以正确形态进入副屏。

### 用户可见效果

合入这个改动以后：

- Android 次级用户 / 分身空间中的应用可以出现在启动列表里
- 用户可以对单个应用单独勾选“横屏副屏启动”
- 为这个应用创建的桌面快捷方式会自动保留横屏参数
- 同一个应用可以按照不同 `userId` 同时运行在多个虚拟显示中
- 应用启动下拉在首次打开时不再容易出现偏移或内容显示不全

### 已验证场景

#### 1. 同一应用在不同用户空间中分别启动到不同副屏

已经确认同一个包名可以同时运行成：

- `com.miHoYo.hkrpg/u15a313`
- `com.miHoYo.hkrpg/u16a313`
- `com.miHoYo.hkrpg/u17a313`

并且每个实例都可以分配到不同的虚拟显示中。

#### 2. 横屏副屏创建成功

已经确认新的横屏路径创建出的虚拟显示为：

```text
2340 x 1080
```

而不是默认的竖屏：

```text
1080 x 2340
```

实际启动后的目标 Activity 也已经确认运行在这些横屏副屏中。

#### 3. 桌面快捷方式兼容

已经确认通过打包版应用使用下面这组参数启动时：

```text
--device-id=...
--package-name=...
--user-id=...
--activity=...
--landscape=1
```

可以正确创建横屏副屏，并把目标次级用户应用启动进去。

### 当前已知限制

目前还存在一个旧的单实例时序问题：

如果主程序处于冷启动状态，同时连续触发多个桌面快捷方式启动请求，第一条请求偶尔可能会丢失。

当前观察结果是：

- 如果主程序已经打开，多次快捷方式启动基本是稳定的
- 如果主程序还没启动、而且在启动过程里连续触发多个快捷方式，请求偶尔会有第一条没有被成功接住的情况

这个问题看起来和本次增加的横屏 / 次级用户逻辑不是同一个层面的 bug，更像是当前单实例生命周期转发时机上的已有边界问题。

### 这个改动对上游的价值

这个改动对下面这些用户会比较有帮助：

- 重度使用 Android 分身 / 多用户空间的用户
- 需要同一个应用多开、多副屏运行的用户
- 更依赖桌面快捷方式一键启动的用户
- 希望在不手动修改全局 scrcpy 参数的前提下，给单个横屏应用设置更合理副屏启动方式的用户

### 建议后续可以继续做的方向

1. 进一步完善冷启动时多快捷方式并发进入时的单实例桥接稳定性。
2. 把横屏 / 竖屏状态做成更直观的图标或文字提示。
3. 后续如果需要，也可以扩展成“应用级启动配置”，而不只是单一的横屏选项。
